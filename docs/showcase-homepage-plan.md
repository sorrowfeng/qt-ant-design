# Showcase 首页对标整改计划

## 状态更新（2026-04-30）

本文件保留为 Showcase 首页对标的历史整改计划。当前计划中的主要收尾项已经完成，并已进入状态文档：

- 当前项目总览：`docs/project-status.md`
- Showcase 审计结论：`docs/visual-audit.md` 的 `Homepage Showcase Audit`
- 当前参考页面：`docs/ant-design-reference.html`
- Qt 示例页面：`examples/pages/ShowcasePage.cpp`

已完成事项：

- 官方 `https://ant.design/index-cn` 首页展示区已检查；捕获时存在 React hydration 和资源错误，因此可重复审计改用本地 HTML 参考。
- HTML 和 Qt Showcase 均已改成只显示控件，不包含左侧主题栏、官网背景或营销页面壳层。
- Qt Showcase 使用已移植 Ant 组件承载可见控件，支持主题信号联动。
- `AntColorPicker` 已从模态编辑器改为 AntD-like 下拉浮层。
- `AntWindow` 已补非最大化外框，避免与同色桌面背景融合。

后续 Showcase 工作不再按本计划从头推进，而是按用户实际发现的差异逐项回到 `docs/visual-audit.md` 的单控件截图对比流程。

## 历史结论（整改前）

整改前的 `ExampleWindow::createShowcasePage()` 只能算“把大类控件摆到了差不多的位置”，离目标图还有明显距离，不能视为完成对标。

这不是一个单点问题，而是三层问题叠加：

1. 页面级构图、留白、层次关系不对
2. 已有控件的尺寸、边框、字重、颜色、状态细节没有统一校准
3. 目标图中有两块核心能力当前并没有真实控件支撑，只能临时占位

当前需要做的不是继续零散微调，而是先把需要修改、优化、补能力的内容完整列清楚，再按顺序实施。

## 当前页与目标图的总体差异

### 页面级差异

- 当前页外层是一个明显的大白卡，目标图更像“白色主画布上的轻量展示区”。
- 当前页上下留白过大，导致内容被拉散，目标图更紧凑、更像一张完整的组件拼图。
- 当前页组件宽度和列宽不统一，目标图所有区块的横向节奏更严格。
- 当前页 modal 与下方内容的重叠关系还不够像官方截图，视觉焦点不够稳定。
- 当前页整体“空气感”偏多，但“秩序感”不足；目标图反过来是高度克制、精确排布。

### 视觉语言差异

- 当前页的边框普遍偏实、偏重，目标图边框更轻、更软。
- 当前页很多控件的圆角半径接近但不一致，目标图圆角系统统一得多。
- 当前页文字字重有些偏重，有些偏散；目标图更接近统一的 400/500/600 层次。
- 当前页控件间垂直基线没有完全对齐，目标图的每一行几乎是精确栅格。

### 能力缺口差异

- 目标图第一行右侧本质上是 `Select` 的 `multiple/tags` 形态。
- 目标图第二行右侧本质上是“标签化多值输入容器”，当前并没有真实对应控件。
- 当前页只能用占位框表达这两块，因此就算把其他控件调得很像，也无法完全复刻目标图。

## 目标拆解

要达到目标图，任务应拆成四层：

1. 修正 Showcase 页面构图
2. 校准已有控件的展示态视觉
3. 补齐缺失控件能力
4. 再回到 Showcase 做二次排版收口

## 逐控件审计

以下按首页出现顺序列出每个控件当前问题、需要修改的地方、计划补的能力、依赖关系和验收标准。

## 1. Modal

### 当前状态

- 页面顶部的 `AntModal` 已能显示标题、正文、关闭按钮和确认/取消按钮
- 结构层面可用
- 视觉层面仍不像目标图

### 当前问题

- 弹窗相对页面的垂直位置偏高或偏低时都会破坏整体构图，需要固定到更贴近目标图的位置
- 弹窗宽度、内边距、正文宽度与目标图还差一档
- 标题字重偏重或字号略大时，会显得更像业务弹窗，不像官方展示图
- 正文行宽、行距、段落边界还不够克制
- 右上角关闭按钮的尺寸、灰度、hover 状态都还不够轻
- footer 按钮组和正文之间的纵向间距还需精修

### 计划修改

- 收紧 dialog 内容区左右 padding
- 统一标题字号、字重、行高
- 调整正文最大宽度与换行节奏
- 调整关闭按钮图标尺寸和 hover 灰阶
- 微调 footer 区的按钮间距、底部边距和整体对齐
- 在 Showcase 场景中固定弹窗展示位置，避免受页面容器高度波动影响

### 依赖关系

- 依赖 `AntModal` 本体样式调校
- 依赖 Showcase 页面最终版式定稿

### 验收标准

- 弹窗宽高比、内边距、按钮布局与目标图肉眼高度接近
- 关闭按钮不抢眼，但存在感与目标图一致

## 2. 信息展示横条（当前使用 AntAlert）

### 当前状态

- 使用 `AntAlert` 承载“信息内容展示”
- 可以表达浅蓝色信息区

### 当前问题

- 当前更像标准提示组件，不像目标图里纯展示用途的横条
- 高度偏高
- 边框偏重
- 背景蓝色饱和度、透明度不够贴
- 左右 padding 偏大或偏小都会破坏条形感
- 文本与边框的垂直居中关系还不够准

### 计划修改

- 重新校准 `AntAlert` 在 `info + no icon + no action` 场景下的紧凑高度
- 调整 `info` 背景色和描边色
- 收紧上下 padding，保留略宽的左右 padding
- 校正文本字号、颜色和 baseline

### 可选方案

- 方案 A：直接优化 `AntAlert`
- 方案 B：为 Showcase 单独做一个展示用轻提示容器

### 建议

- 先改 `AntAlert`
- 如果会破坏其他示例页，再考虑抽 Showcase 专用容器

### 验收标准

- 横条一眼接近目标图的高度、蓝度、圆角和文本位置

## 3. 第一行左侧：单选 Select

### 当前状态

- `AntSelect` 单选模式可用
- placeholder、下拉箭头、边框、尺寸都已经具备

### 当前问题

- 控件高度与目标图仍有偏差
- placeholder 灰度偏浅或偏深
- 箭头位置和尺寸还不够精确
- 输入框内容区与边框的垂直居中不够稳
- 控件本身圆角和目标图稍有差异
- 边框颜色在空态下偏重

### 计划修改

- 调整 Select 基础高度
- 调整 paddingX、arrowWidth、fontSize 的组合
- 调整空态边框色和 hover 边框色
- 校正 placeholder 色值
- 校正箭头图标的视觉中心位置

### 验收标准

- 第一行左侧下拉框在空态时与目标图的“高度、箭头、边框、placeholder”接近

## 4. 第一行中间：颜色输入展示块

### 当前状态

- 当前是手拼 `QFrame + QLabel + 色块`
- 只能临时展示形态

### 当前问题

- 不属于可复用控件
- 与其他输入框的边框、圆角、padding 体系没有真正统一
- 色块大小、圆角、左右边距还不够像目标图
- 十六进制文本和左侧色块的对齐还需要微调

### 计划修改

- 抽成独立的 Showcase 组件块
- 与 `AntSelect` / `AntDatePicker` 使用同一套输入壳层参数
- 固定色块大小、圆角和间距
- 调整文本字重与颜色

### 后续演进

- 如果后面要做真正的颜色输入控件，可把这个块迁回控件层

### 验收标准

- 看起来就是官方图里那个颜色输入容器，而不是临时拼的盒子

## 5. 第一行右侧：Select multiple/tags

### 当前状态

- 当前没有真实能力
- 只能用占位框 + `AntTag` 模拟

### 当前缺失能力

- `multiple`
- `tags`
- 输入框与 tag 混排
- 动态 placeholder
- tag 删除按钮
- 多选容器内自动换行或最少单行布局
- 多值输入与 popup 选择联动

### 当前问题

- 占位文本过于直接，破坏整体观感
- tag 高度、关闭图标、间距还不是最终态
- 容器看起来不是“真实可交互控件”

### 计划修改

#### 第一阶段：占位优化

- 做统一占位容器
- 降低提示文本存在感
- 让占位容器在视觉上尽量像真实控件壳层

#### 第二阶段：真实能力补齐

- 在 `AntSelect` 中新增 `selectionMode`
- 支持 `single / multiple / tags`
- 输入区改为 token 容器
- 支持 tag 删除
- 支持 placeholder 与 tag 混排
- 支持 popup 勾选多项
- 支持键盘回删删除最后一个 tag

### 依赖关系

- 强依赖 `AntTag` 的紧凑样式
- 强依赖 `AntSelect` popup 与输入壳层重构

### 验收标准

- 最终必须由真实 `AntSelect` 多选模式替换当前占位

## 6. 第二行左侧：DatePicker

### 当前状态

- `AntDatePicker` 单值模式可用
- 首页里能显示空态输入框

### 当前问题

- 控件壳层高度与第一行输入不完全对齐
- 日历图标位置需要更精确
- 空态 placeholder 颜色和目标图还有偏差
- 图标右侧 padding 和边框圆角需要进一步统一
- 当前和第一行其他输入框之间的壳层语言还没完全一致

### 计划修改

- 统一 DatePicker 与 Select 的壳层参数
- 调整图标区域宽度与图标实际绘制位置
- 调整空态文字颜色和字体基线
- 校正 hover/focus 态边框

### 验收标准

- 第二行左侧 DatePicker 与目标图在空态时近似

## 7. 第二行右侧：标签化多选日期 / 复合输入

### 当前状态

- 当前无真实控件
- 使用占位容器 + `AntTag` 表达

### 目标形态本质

- 一个输入容器
- 容器内已有 token
- 还保留输入/占位区域
- 后续可接日期选择或其他值来源

### 当前缺失能力

- token 化多值展示
- 复合输入容器
- token 删除
- 占位与 token 共存
- 与日期选择逻辑的真实联动

### 计划修改

#### 方案一：扩展 `AntDatePicker`

- 增加多值模式
- 增加 token 渲染
- 增加多值数据结构

#### 方案二：新建组合控件

- 输入容器负责 token
- 日期弹层作为数据来源
- 与 `AntSelect tags` 共用 token 容器能力

### 建议

- 优先做 token 输入容器基础层
- 再决定挂到 `AntDatePicker` 还是新控件

### 验收标准

- 最终必须由真实复合输入控件替换当前占位

## 风险

- 如果直接在首页用大量局部 stylesheet 修，会把问题藏起来，不会真正提升组件质量
- 如果先补复杂能力而不先收紧版式，截图仍然会“不像”
- 如果先只改首页不动组件本体，其他页面和未来场景会继续不统一
